# **Задание 29**

**OAuth 2.0** - это протокол авторизации, который позволяет приложениям получать ограниченный доступ к пользовательским данным без раскрытия паролей. OAuth играет ключевую роль в безопасном предоставлении доступа к защищенным ресурсам.

--- 

Особенности использования OAuth в API:

OAuth обеспечивает делегированный доступ — третьи стороны получают только конкретные разрешения (scopes), а не полный контроль над аккаунтом.
Протокол поддерживает несколько потоков авторизации (grant types): Authorization Code для веб-приложений, Client Credentials для сервисного взаимодействия, Implicit для одностраничных приложений.
Токены доступа (access tokens) имеют ограниченное время жизни и scope, что снижает риски при компрометации.
Refresh tokens позволяют обновлять access tokens без повторного взаимодействия с пользователем.

---

Риски безопасности при использовании OAuth в API:

Перехват authorization code — злоумышленник может перехватить код авторизации, если используется ненадежный redirect_uri или отсутствует PKCE.
Утечка токенов — access tokens могут попасть к злоумышленнику через небезопасное хранение, логирование или передачу по незащищенным каналам.
Подделка межсайтовых запросов (CSRF) — атакующий может подменить параметры авторизации и перенаправить токен на свой сервер.
Небезопасная валидация redirect_uri — если сервер не проверяет точное соответствие redirect_uri, возможен открытый редирект.
Чрезмерные разрешения (over-privileged scopes) — приложения могут запрашивать больше прав, чем необходимо, что увеличивает ущерб при компрометации.
Небезопасное хранение client_secret — в публичных клиентах (мобильные приложения, SPA) секрет может быть извлечен из кода.

----

Меры защиты:

Всегда использовать PKCE для предотвращения перехвата кода авторизации.
Строгая валидация redirect_uri — проверять полное соответствие с заранее зарегистрированными URI.
Короткое время жизни access tokens и безопасное хранение refresh tokens.
Обязательное использование HTTPS для всех коммуникаций.
Регулярный аудит выданных разрешений и отзыв неиспользуемых токенов.
Применение принципа минимальных привилегий при запросе scopes.

Правильно реализованный OAuth обеспечивает безопасное взаимодействие между приложениями и API, но требует постоянного внимания к деталям реализации и поддержания безопасности на всех этапах работы с токенами.

## 8. Реализация и мониторинг
**Внедрение и контроль:**
- Внедрение выбранных мер защиты
- Регулярный мониторинг эффективности контролей
- Периодический пересмотр оценки рисков
- Обновление оценки при изменениях в бизнес-процессах или инфраструктуре
---
## 9. Документирование и отчетность
**Формирование результатов:**
- Составление отчета об оценке рисков
- Ведение реестра рисков
- Документирование принятых мер
- Отчетность перед руководством и регуляторами
---
**Ключевые принципы:**
- Оценка должна быть непрерывным процессом, а не разовым мероприятием
- Необходимо учитывать как технические, так и человеческие аспекты
- Важен баланс между безопасностью и удобством бизнес-процессов
- Регулярный пересмотр при изменении законодательства или бизнес-среды
---
